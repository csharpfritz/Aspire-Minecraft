## Quick Start: 60 Seconds to Your First Minecraft Dashboard

### Prerequisites

- [.NET 10 SDK](https://dotnet.microsoft.com/)
- [Docker Desktop](https://www.docker.com/products/docker-desktop/)
- Minecraft Java Edition (to connect and see the world)

### 1. Install the package

```shell
dotnet add package Fritz.Aspire.Hosting.Minecraft --version 0.1.0
```

### 2. Wire it up in your AppHost `Program.cs`

```csharp
using Aspire.Hosting.Minecraft;

var builder = DistributedApplication.CreateBuilder(args);

var redis = builder.AddRedis("cache");
var api = builder.AddProject<Projects.MyApi>("api");

var minecraft = builder.AddMinecraftServer("minecraft", gamePort: 25565, rconPort: 25575)
    .WithBlueMap(port: 8100)
    .WithOpenTelemetry()
    .WithAspireWorldDisplay<Projects.Aspire_Hosting_Minecraft_Worker>()
    .WithParticleEffects()
    .WithTitleAlerts()
    .WithWeatherEffects()
    .WithBossBar()
    .WithSoundEffects()
    .WithMonitoredResource(api)
    .WithMonitoredResource(redis);

builder.Build().Run();
```

Every `.With*()` method is opt-in. Start with just `AddMinecraftServer()` and layer on features as you like.

### 3. Run and connect

```shell
dotnet run
```

Open Minecraft Java Edition ‚Üí Add Server ‚Üí `localhost:25565` ‚Üí join and explore!

> **Tip:** The first launch downloads the Paper server image (~1 GB) and generates the world. Give it 60‚Äì90 seconds on the first boot.

## Feature Tour: What Ships in v0.1.0

### üèóÔ∏è Minecraft Server as a First-Class Aspire Resource

`AddMinecraftServer()` spins up a [Paper](https://papermc.io/) server in Docker via the trusted [`itzg/minecraft-server`](https://hub.docker.com/r/itzg/minecraft-server) image. It appears in the Aspire dashboard with game, RCON, and BlueMap endpoints. Other services can `.WaitFor()` the Minecraft server just like any database or cache.

```csharp
var minecraft = builder.AddMinecraftServer("minecraft");
// That's it. You have a Minecraft server managed by Aspire.
```

### üìä OpenTelemetry ‚Äî JVM Meets .NET

`.WithOpenTelemetry()` injects the OpenTelemetry Java agent into the Minecraft server container. JVM metrics (heap, GC, threads, CPU) flow into the Aspire dashboard alongside your .NET services. Game metrics (TPS, MSPT, player count) are polled via RCON and exported as custom OTEL metrics.

| Metric | What It Tells You |
|---|---|
| `jvm.memory.used` | Is the server running out of heap? |
| `minecraft.tps` | Is the server keeping up? (target: 20.0) |
| `minecraft.mspt` | How long each tick takes |
| `minecraft.players.online` | Who's connected |
| `minecraft.rcon.latency_ms` | RCON round-trip health |

### üó∫Ô∏è BlueMap ‚Äî A 3D Web Map in Your Dashboard

`.WithBlueMap()` installs the [BlueMap](https://bluemap.bluecolored.de/) plugin and exposes its web UI as an HTTP endpoint in the Aspire dashboard. Click it and you get a full 3D, interactive map of your Minecraft world ‚Äî including the Aspire resource structures.

### üèõÔ∏è In-World Health Display

`.WithAspireWorldDisplay<TWorker>()` registers a background worker that connects to the Minecraft server via RCON. Each `.WithMonitoredResource()` call creates:

- **A torch-topped structure** ‚Äî emerald blocks for healthy, redstone blocks for unhealthy
- **A floating hologram** ‚Äî resource name and current status, updated in real time
- **A scoreboard entry** ‚Äî sidebar scoreboard showing all resources at a glance
- **Player chat alerts** ‚Äî health changes announced as they happen

### üå©Ô∏è Boss Bar Health Meter

`.WithBossBar()` adds a persistent bar at the top of the screen showing overall fleet health as a percentage. The color shifts with your system state:

- **Green** (‚â• 75% healthy) ‚Äî smooth sailing
- **Yellow** (25‚Äì74% healthy) ‚Äî degraded
- **Red** (< 25% healthy) ‚Äî critical

### ‚õàÔ∏è Weather = System Health

`.WithWeatherEffects()` links Minecraft weather to your fleet's overall health ratio:

- **Clear skies** ‚Äî 100% of services healthy
- **Rain** ‚Äî some services degraded (‚â• 50%)
- **Thunderstorm** ‚Äî critical state (< 50%)

Weather only changes on state transitions, not every poll cycle ‚Äî so the sky stays stable.

### üì¢ Title Screen Alerts

`.WithTitleAlerts()` fires dramatic full-screen Minecraft title text on health transitions:

- **Red "‚ö† SERVICE DOWN"** when a resource goes unhealthy
- **Green "‚úÖ BACK ONLINE"** when it recovers

These are impossible to miss. That's the point.

### üîä Sound Effects

`.WithSoundEffects()` plays Minecraft sounds on health state changes:

- **Wither ambient growl** when a service goes down
- **Level-up chime** when a service recovers
- **Challenge complete fanfare** when all services return to green

### ‚ú® Particle Effects

`.WithParticleEffects()` adds visual flair at resource structures:

- **Smoke + flame particles** billow from a structure when its service goes down
- **Happy villager particles** dance above it when it recovers

## How It Works: Under the Hood

```
Your AppHost (Program.cs)
  ‚îÇ
  ‚îú‚îÄ‚îÄ builder.AddMinecraftServer("minecraft")
  ‚îÇ     ‚Üí Docker: itzg/minecraft-server (Paper)
  ‚îÇ     ‚Üí Ports: 25565 (game), 25575 (RCON), 8100 (BlueMap)
  ‚îÇ     ‚Üí OTEL Java Agent ‚Üí Aspire Dashboard
  ‚îÇ
  ‚îú‚îÄ‚îÄ .WithAspireWorldDisplay<Worker>()
  ‚îÇ     ‚Üí .NET Worker Service (child resource)
  ‚îÇ     ‚Üí RCON commands ‚Üí Minecraft server
  ‚îÇ     ‚Üí Polls resource health via HTTP/TCP
  ‚îÇ     ‚Üí Publishes minecraft.* OTEL metrics
  ‚îÇ
  ‚îî‚îÄ‚îÄ Your Services (API, Web, Redis, Postgres‚Ä¶)
        ‚Üí Monitored via .WithMonitoredResource()
        ‚Üí Visualized as in-world structures
```

**RCON (Remote Console)** is the Minecraft server's built-in admin protocol. The worker service speaks RCON to execute commands like `/bossbar set`, `/weather`, `/title`, `/particle`, and `/playsound`. It's the same protocol that production server admins use ‚Äî reliable, low-overhead, and well-documented.

**The Worker Service** is a standard .NET `BackgroundService` that runs as a child resource of the Minecraft server in Aspire. It discovers monitored resources from environment variables injected by `WithMonitoredResource()`, polls their health endpoints, and translates health state into RCON commands. Each feature (boss bar, weather, particles, etc.) is opt-in via environment variables set by the corresponding `.With*()` extension method.

**State tracking** prevents command spam. The worker remembers the last weather state, boss bar value, and per-resource health so it only sends RCON commands when something actually changes.
