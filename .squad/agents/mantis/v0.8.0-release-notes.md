# Aspire.Hosting.Minecraft v0.8.0 Release Notes

We're excited to announce v0.8.0 of Aspire.Hosting.Minecraft — bringing motion and life to your distributed system! This release transforms your resource village from a static monitoring dashboard into a living, breathing infrastructure where minecarts travel along rails between dependent services and NPC villagers run a bustling fruit stand at the town center.

## 🎯 What's New

### Minecart Rail Network

Your dependencies now move. Enable `.WithMinecartRails()` to build a powered rail network that physically connects dependent resources with L-shaped rail paths. Chest minecarts travel from dependent services to their parent dependencies, visualizing the flow of your system architecture in real-time.

**How it works:**
- Each building gets an E-W rail segment running along its back edge
- Rails curve south at junctions, connecting to dedicated N-S trunk lines
- Powered rails maintain minecart speed; regular rails at junctions enable auto-curving
- Rails automatically navigate around buildings and cross canals via elevated bridges
- Minecarts spawn dynamically as services come online and circulate through the network

**Why it matters:** Understanding distributed system dependencies is hard. When you see a minecart physically traveling from your web frontend to your API service, then from the API to Redis, the dependency chain becomes tangible. No more staring at configuration files — just watch the rails.

```csharp
builder.AddMinecraftServer("minecraft")
    .WithAspireWorldDisplay<Projects.Worker>()
    .WithMinecartRails()  // Enable the rail network
    .WithMonitoredResource(api)
    .WithMonitoredResource(redis);
```

### Fruit Stand with NPC Villagers

A new easter egg feature adds five named NPC villagers running a fruit stand at the center of your village. Maddy and Damian work behind the counter, Fowler greets customers up front, and Brady and Scott browse nearby. These villagers are persistent, named, and have proper professions — they'll wander the village naturally, bringing a sense of community to your infrastructure town.

**Structure details:**
- 5×3 market stall with spruce plank floor and orange wool awning
- Oak slab counter, barrel storage, and hanging lantern
- "Fresh Fruit! / Maddy & Co." signage
- Positioned dynamically at the true village center based on resource count

**The villagers:**
- **Maddy** (farmer) — behind the counter
- **Damian** (butcher) — behind the counter  
- **Fowler** (fisherman) — greeting customers
- **Brady** (librarian) — browsing the stand

This feature is always enabled when using `.WithAspireWorldDisplay()` — no opt-in needed. Look for the fruit stand between your village quadrants!

## 🔧 Improvements

### Error Minecart System Enhancements

Building on the error boat visualization from earlier versions, this release adds proper buffering and canal readiness gating:

- **Buffered spawning:** Error minecarts triggered before canal infrastructure is ready are now queued and spawned automatically once canals are built
- **Accurate positioning:** Fixed spawn coordinates to match actual per-building canal locations (behind each structure at `oz + StructureSize + 4`)
- **Motion physics:** Error boats now spawn with westward velocity (`{Motion:[-0.5,0.0,0.0]}`) for immediate propulsion through the canal system
- **Service coordination:** ErrorBoatService properly depends on CanalService, ensuring infrastructure is ready before spawning entities

### Aspire Dashboard Integration

The GrandVillageDemo sample now includes a "Trigger Error" button in the Aspire dashboard:

- Click the button on the API service to simulate a failure (HTTP 503)
- Aspire health monitoring detects the unhealthy state
- Minecraft worker spawns an error boat in the canal system
- Perfect for demoing the error visualization pipeline

Uses the new `.WithHttpCommand()` pattern introduced in .NET Aspire 13.1 for clean dashboard-to-resource integration.

### Code Quality & Architecture

- **Canal system refactoring:** CanalService now uses ground-level rail placement (SurfaceY + 1) instead of trenched canals, reducing RCON command overhead
- **Junction curve logic:** Rail junctions use regular rails (not powered) for proper auto-curving when minecarts change direction
- **Bridge clearance:** Rails and bridges properly coordinate to avoid collision with canal infrastructure
- **Forceload coverage:** All canal routes are now covered by chunk forceloading to prevent entity despawning during transit
